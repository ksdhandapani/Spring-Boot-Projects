Challenges related to Container Orchestration
---------------------------------------------
---------------------------------------------

Sometimes we may have more than 100 microservices, which means we may have more than 100 different containers running on our production servers. When we have such a large number of containers, we need a component to handle container orchestration. Just as a conductor in a musical orchestra coordinates many musicians, we need a component that can control our containers based on our requirements.

Challenge 1: How are we going to automate the deployments, rollouts and rollbacks?
----------------------------------------------------------------------------------

How do we automate the deployment of containers into a complex cluster environment and perform rollouts of new versions of the containers without downtime, along with an option for automatic rollback in case of any issues?

What is rollout and rollback?

Whenever we try to make changes or build a new version of our microservices, we need to push them or deploy them to a production server. When we try to do that deployment, larger organizations cannot afford downtime. So in such scenarios, they will go with the option of rollout. As part of this rollout, they replace the containers one by one with the latest Docker image. 

For the ms-accounts microservice, we have three Docker containers running in production. With the help of this rollout, we can first create a new container with the latest Docker image. Once the new container is available, we can then terminate the previously running containers. This will avoid any downtime that organizations may face during the deployment. 

On a similar line, think of when we roll out a new feature or a new Docker image into our containers, and we face an issue, or a bug is identified in the production server. In such scenarios, we should also have the flexibility of automatic rollback to the previously running version of the Docker image in case of any issues. 

Challenge 2: How are we going to make our microservices self-healing capable?
-----------------------------------------------------------------------------

How do we automatically restart containers that fail, replace containers, kill containers that don't respond to our user-defined health check, and do not advertise them to clients until they are ready to serve?

That means if one of the microservice containers is not responding properly or it is responding very slowly, we need to have someone to regularly perform the health check on these running containers and take an automatic decision by killing the containers that are not responding properly and replacing them with new containers.

Challenge 3: How are we going to make our microservices auto-scalable?
----------------------------------------------------------------------

How do we monitor our services and scale them automatically based on metrics like CPU Utilization, etc?

So what is scaling? When are we seeing the increase in traffic that is coming towards our application? We can try to onboard a larger number of containers or more number of services, so that is vertical scaling, but doing that manually for each and every microservice is going to be a bsuper super impossible task. So that is why we need to look for options on how to do auto scaling of our microservices based upon the CUP utilization or other metrics of the running containers.

We should be able to automatically make a decision whether to scale up or scale down the running containers. 

Example: Let's take an example of Netflix, usually there will be a lot of traffic to Netflix on Friday night, on the Staturday and Sunday. So during these days the Netflix should automatically scale their number of microservice instances or applications so that it can stream the content to its customers without any issues. And similarly, there can be a sudden spike in traffic sometimes, maybe due to some long weekend or due to some holiday in the middle of the week. So, in all such scenarios, Netflix should be able to automatically handle that by increasing the number of microservices at runtime with the help of auto scaling. 

Very similarly, we should also be able to perform auto scaling on our microservices.

Challenges Related to Container Orchestration
---------------------------------------------
---------------------------------------------

In a real-world microservices architecture, we may have hundreds of microservices, which means hundreds of containers running in production.

Managing such a large number of containers manually is not possible. So we need a container orchestration component to manage them efficiently.

Just like a music conductor controls and coordinates multiple musicians, a container orchestration system controls containers based on our requirements such as deployment, health, scaling, and availability.

Challenge 1: Automating Deployments, Rollouts, and Rollbacks
------------------------------------------------------------

One of the major challenges is how to automate deployments in a complex cluster environment without downtime, and also support automatic rollback if something goes wrong.

>> What is Deployment?

Deployment means pushing a new version of a microservice (Docker image) to the production environment.

>> What is Rollout?

In large organizations, downtime is not acceptable. So instead of stopping all containers and starting new ones, we use rolling deployment (rollout).

In rollout:

- Containers are replaced one by one
- New containers are started with the latest Docker image
- Old containers are terminated only after the new ones are ready

Example:

Assume the ms-accounts microservice has 3 containers running in production.

- A new container with the latest image is started
- Once it is healthy, one old container is stopped
- This process continues until all containers are upgraded
- This approach ensures zero downtime during deployment.

>> What is Rollback?

Sometimes after deploying a new version:

- A bug is detected
- Performance issues occur
- Production errors are reported

In such cases, we should be able to automatically rollback to the previous stable version of the Docker image.

Rollback ensures:

- Faster recovery
- Minimal business impact
- High system reliability

Challenge 2: Making Microservices Self-Healing
----------------------------------------------

Microservices should be self-healing, meaning they can recover from failures automatically without manual intervention.

Self-healing includes:

- Automatically restarting failed containers
- Replacing crashed containers
- Killing containers that fail health checks
- Not routing traffic to containers until they are fully ready

If a container:

- Stops responding
- Becomes very slow
- Fails a user-defined health check

Then the system should:

- Continuously perform health checks
- Automatically terminate unhealthy containers
- Create new containers to replace them
- Ensure only healthy containers serve user requests

This improves system stability and availability.

Challenge 3: Making Microservices Auto-Scalable
-----------------------------------------------

Traffic to applications is not constant. It can increase or decrease based on time, usage, or events.

Auto-scaling helps microservices handle traffic changes automatically.

Auto-scaling involves:

- Monitoring metrics like CPU utilization, memory usage, or request count
- Automatically scaling up or scaling down containers based on load
- Manually scaling containers for each microservice is not practical. So we need an automated mechanism.
- In microservices, scaling is usually horizontal scaling (adding or removing containers), not vertical scaling.

Example:

Netflix experiences high traffic on:

- Friday nights
- Saturdays and Sundays
- Holidays and long weekends

During these times:

- Netflix automatically adds more microservice instances
- This ensures smooth streaming for users

When traffic reduces:

- Extra instances are removed
- Resources are saved

Similarly, our microservices should:

- Scale up during high traffic
- Scale down during low traffic
- Handle sudden traffic spikes automatically

>> Summary

- Container orchestration is required to manage large numbers of containers
- Rollouts ensure zero-downtime deployments
- Rollbacks provide safety during failed deployments
- Self-healing ensures high availability
- Auto-scaling ensures performance and cost efficiency

- Its own OS
- Application
- Required libraries and dependencies

Problems with VMs:

- Heavy and slow to start
- High resource usage because each VM runs a full OS
- Scaling is manual and slow
- Managing many VMs becomes complex
- VMs are good for strong isolation, but not ideal for microservices.

>>> Docker (Containers)

Docker introduced containers, which are lightweight compared to VMs.

In Docker:

- Containers share the host OS

Each container has:

- Application
- Required libraries
- No separate OS per container

Advantages of Docker:

- Fast startup
- Less resource usage
- Easy to package and ship applications
- Works well for microservices

Limitations of Docker alone:

No built-in:

- Auto-scaling
- Self-healing
- Load balancing

Managing containers manually becomes difficult at scale

Docker solves packaging, but not production management.

Kubernetes is a container orchestration platform.

Kubernetes manages Docker containers (or other containers) at scale.

What Kubernetes provides:

- Automated deployments
- Rolling updates and rollbacks
- Self-healing (restart failed containers)
- Auto-scaling based on CPU or other metrics
- Service discovery and load balancing
- Configuration and secret management

Kubernetes is used when:

- We have many containers

>> What is a Cluster?

A cluster is a group of physical or virtual machines that work together to deliver applications reliably.
In a Kubernetes cluster, multiple machines cooperate to ensure high availability, scalability, and self-healing of microservices.

>> Why Not Docker Compose?

Docker Compose is designed for single-host deployments.
In real-world production systems, applications may consist of hundreds of microservices that need to run across multiple servers.

Docker Compose lacks:

- Multi-node scheduling
- Self-healing across nodes
- Rolling updates
- Fault tolerance

Kubernetes provides these features and acts as a container orchestration platform.

We can interact with Kubernetes using:

- kubectl CLI (primary and official tool)
- Optional web dashboards (e.g., Kubernetes Dashboard, Lens)

>> Kubernetes Cluster

A Kubernetes cluster consists of two types of nodes:

1. Control Plane (Master Node)

Responsible for:

- Managing cluster state
- Scheduling workloads
- Maintaining desired state

The master node is responsible for managing an entire cluster. It monitors the health check for all the nodes in the cluster, stores members' information regarding different nodes, plans the containers that are scheduled to certain worker nodes, monitors containers and nodes, etc. So when a worker node fails, the master moves that workload from the failed node to another healthy worker node. 

Below are the details of the four basic components present inside the control plane:

> API Server: The API Server is the primary interface for interacting with the Kubernetes cluster. It exposes the Kubernetes API, which allows users and other components to communicate with the cluster. All administrative operations and control commands are sent to the API server, which then processes and validates them.

> Scheduler: The scheduler is responsible for placing Pods onto available nodes in the cluster. It takes into account factors like resource requirements, affinity, anti-affinity, and other constraints to make intelligent decisions about which node to assign a Pod to. The scheduler continuously monitors the cluster and ensures that Pods are distributed optimally.

> Controller Manager: The controller manager maintains the cluster. It handles node failures, replicates components, maintains the correct number of pods, etc. It constantly tries to keep the system in the desired state by comparing it with the current state of the system.

> etcd: etcd is a distributed key-value store that serves as the cluster's primary data store. It stores the configuration data and the desired state of the system, including information about Pods, Services, ReplicationControllers, and more. The API Server interacts with etcd to read and write cluster data. 

2. Worker Nodes

Responsible for:

- Running application Pods
- Executing container workloads
- Serving traffic routed to Pods
- Interacting with the Cluster

A worker node is a physical server or a virtual machine running either on-premises or in the cloud.
Any machine capable of running a CRI-compatible container runtime can function as a worker node.

Worker nodes:

- Provide compute, storage, and networking resources
- Run application workloads in the form of Pods
- Perform the heavy-lifting in a Kubernetes cluster
- Workloads are scheduled to worker nodes by the Kubernetes control plane, enabling fault tolerance, replication, and scalability across the cluster.

> Pods: Pods are the smallest deployable unit in Kubernetes, just as containers are the smallest deployable unit in Docker.

A Pod:

- Consists of one or more containers
- Provides a shared network namespace and storage
- Represents a single logical application instance
- Pods are not virtual machines.
- They are lightweight abstractions used to group tightly coupled containers.
- Each time a Pod is created, it receives a unique IP address from the cluster’s Pod networking solution.

> Sidecar Pattern

Sometimes an application container needs a helper container (logging, monitoring, proxy, etc.).

- Such helper containers run alongside the main container in the same Pod.
- This design is called the sidecar pattern.

Components of a Worker Node

> Kubelet: Kubelet is an agent that runs on each worker node and communicates with the control plane. It receives Pod specifications and ensures that the containers defined in those Pods are running and healthy.

Responsibilities include:

- Starting and stopping containers
- Monitoring container health
- Reporting node and Pod status to the API Server

> kube-proxy: kube-proxy runs on each node and implements part of the Kubernetes Service abstraction.

It:

- Maintains networking rules using iptables or IPVS
- Routes Service traffic to the appropriate Pod IPs
- Enables internal cluster communication
- kube-proxy does not expose applications externally by itself.

kube-proxy is responsible for service networking inside the cluster.

- It implements Kubernetes Services
- Routes traffic to the correct Pod IPs
- Uses iptables or IPVS rules
- kube-proxy does not expose applications to the outside world directly.

External exposure is done by:

- Service type LoadBalancer
- NodePort
- Ingress Controller

> Container Runtime: The container runtime is responsible for pulling images and running containers on a worker node.

Kubernetes interacts with the runtime via the Container Runtime Interface (CRI).

Common runtimes include:

- containerd
- CRI-O

Docker is no longer used directly by Kubernetes.

>>> Kubernetes Control Plane (Master Node)
------------------------------------------
------------------------------------------

The Control Plane is responsible for managing the cluster and maintaining its desired state.
It consists of the following components:

- API Server
- etcd
- Controller Manager
- Scheduler

>> API Server
-------------

The API Server is the central communication hub of Kubernetes.
All cluster interactions go through the API Server:

- kubectl
- Controller Manager
- Scheduler
- Kubelets on worker nodes

It:

- Validates and processes REST requests
- Persists cluster state in etcd
- Acts as the single source of interaction for all components

>> etcd
-------

etcd is a distributed key-value store and the source of truth for the Kubernetes cluster.

It stores:

-Desired state (Pod specs, Deployment replicas, configs)
- Cluster configuration
- Metadata
- Secrets and ConfigMaps

Example stored state:

A Deployment requires 2 replicas
Image: simplybyte/calculator
Container port: 8080

>> Controller Manager
---------------------

The Controller Manager runs multiple controllers that continuously ensure:

Desired State == Actual State

Examples of controllers:

- Deployment Controller
- ReplicaSet Controller
- Node Controller
- Endpoint Controller

Desired State:- What the user declares (e.g., replicas = 2)
Actual State:- What is currently running in the cluster

- If a mismatch is detected, the controller takes corrective action via the API Server.
- Controllers manage Kubernetes resources (Pods, Nodes, ReplicaSets) — not containers directly.

>> Scheduler
------------

The Scheduler decides which worker node a Pod should run on.

It evaluates:

- CPU and memory availability
- Resource requests and limits
- Node affinity / anti-affinity
- Taints and tolerations

Once a suitable node is found, the Scheduler binds the Pod to that node.
The Scheduler does not create Pods or containers — it only assigns Pods to nodes.

>>> Worker Nodes
-----------------
-----------------

- A Kubernetes cluster can have multiple worker nodes.
- Each worker node runs application workloads.

Each worker node contains:

- Kubelet
- Container Runtime (containerd / CRI-O)
- kube-proxy
- Pods

>> Pods (Fundamental Unit)
--------------------------

A Pod is the smallest deployable unit in Kubernetes.

A Pod can contain:

- One container (most common)
- Multiple tightly coupled containers (Group tightly coupled containers)

Containers in a Pod:

- Share the same IP address
- Share storage volumes
- Communicate via localhost

>> Kubelet
----------

Kubelet is the node agent running on each worker node.

Responsibilities:

- Watches for Pods assigned to its node (via API Server)
- Ensures containers in the Pod are running
- Reports Pod and Node status back to the API Server

Flow:

- API Server informs the Kubelet about a new Pod
- Kubelet reads the Pod specification
- Kubelet instructs the container runtime to start containers

>> Container Runtime & CRI
--------------------------

Kubernetes uses the Container Runtime Interface (CRI) to interact with container runtimes.

Common runtimes:

- containerd
- CRI-O

Responsibilities:

- Pull container images
- Create and delete containers
- Manage container lifecycle
- Docker is no longer used directly in Kubernetes (deprecated since v1.24).

>> kube-proxy
-------------

kube-proxy manages networking and service routing on each node.

It:

- Implements Kubernetes Services
- Uses iptables or IPVS rules
- Routes traffic to the correct Pod IPs
- kube-proxy is not a traditional proxy; it programs networking rules.

>>> Self-Healing Flow (End-to-End)
----------------------------------

A Pod crashes on a worker node
Kubelet detects the failure
Kubelet reports the status to the API Server
Controller Manager detects a mismatch between desired and actual state
Controller requests creation of a replacement Pod
Scheduler selects a suitable node
API Server assigns the Pod to that node
Kubelet creates containers using the container runtime
This continuous reconciliation enables Kubernetes self-healing.

Additional Important Components

>> Container Network Interface (CNI)
------------------------------------

Kubernetes relies on CNI plugins (Calico, Flannel, Cilium) for networking.

Responsibilities:

- Assign Pod IP addresses
- Enable Pod-to-Pod communication
- Enforce network policies

>> Cloud Controller Manager (Cloud Environments)
------------------------------------------------

Manages cloud-specific resources:

- Load balancers
- Volumes
- Node lifecycle

Keeps cloud logic separate from core Kubernetes components

>> Key Takeaways
-----------------

- Kubernetes is declarative and self-healing
- Control Plane maintains desired state
- Worker nodes run Pods
- Pods are the fundamental unit (not containers)
- Continuous reconciliation ensures reliability and scalability

Step 3: Create Environment Variables in the Kubernetes Cluster Using ConfigMaps

References:

- [Kubernetes ConfigMaps Documentation](https://kubernetes.io/docs/concepts/configuration/configmap/)
- [Kubernetes Secrets Documentation](https://kubernetes.io/docs/concepts/configuration/secret/)

Problem:

In our current Docker Compose files, all containers rely on several environment variables. These include the active profile (SPRING_PROFILES_ACTIVE), the configuration import URL (SPRING_CONFIG_IMPORT), and the Eureka Server URL (EUREKA_CLIENT_SERVICEURL_DEFAULTZONE).

Solution:

Since all our microservices depend on these environment variables to function, we need to find a way to create similar variables within the Kubernetes Cluster. To do this, we will create a ConfigMap object in Kubernetes.

A ConfigMap is an object used to store non-confidential data in key-value pairs. Pods can consume ConfigMaps as environment variables, command line arguments, or as configuration files in a volume.

Note: ConfigMaps do not provide secrecy or encryption. If you need to store confidential data, use a Secret instead of a ConfigMap or employ additional tools to keep your data secure.

Step 3.1: Create a File Named 'configmaps.yaml' with the below content

In the 'configmaps.yaml' file, we will not define any properties related to Kafka, RabbitMQ, or OpenTelemetry. The reason for this is that we will cover how to set up Kafka or Grafana components in Kubernetes when we discuss Helm Charts. Writing Kubernetes manifest files for industry-provided components like Kafka and Grafana can be unnecessary and overly complicated.

For now, we will deploy only our microservices along with KeyCloak. KeyCloak is essential; we cannot skip it because our gateway server is secured. Without the KeyCloak server, we will not be able to access any APIs necessary for creating data within our microservices.

	configmaps.yaml

		apiVersion: v1
		kind: ConfigMap
		metadata:
		  name: xyzbank-configmap
		data:
		  SPRING_PROFILES_ACTIVE: prod
		  SPRING_CONFIG_IMPORT: "configserver:http://ms-config-server:8071/" # Location of the Config Server
		  EUREKA_CLIENT_SERVICEURL_DEFAULTZONE: "http://ms-eurekaserver:8070/eureka/"
		  CONFIG_SERVER_APPLICATION_NAME: "ms-config-server"
		  EUREKASERVER_APPLICATION_NAME: "ms-eurekaserver"
		  GATEWAY_SERVER_APPLICATION_NAME: "ms-gateway-server"
		  ACCOUNTS_APPLICATION_NAME: "ms-accounts"
		  LOANS_APPLICATION_NAME: "ms-loans"
		  CARDS_APPLICATION_NAME: "ms-cards"
		  KEYCLOAK_BOOTSTRAP_ADMIN_USERNAME: "admin"
		  KEYCLOAK_BOOTSTRAP_ADMIN_PASSWORD: "admin"
		  SPRING_SECURITY_OAUTH2_RESOURCESERVER_JWT_JWK-SET-URI: "http://keycloak:8080/realms/master/protocol/openid-connect/certs" # With this path, we are telling our Gateway Server to validate the JWT tokens by using the public keys exposed by Keycloak server. The KeyCloak Server has Private Certificate or Key, using the private key only the KeyCloak issues the new tokens.
